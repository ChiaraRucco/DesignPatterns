RQ1
Pattern Introduction

MIND

Metadata-Driven Ingestion Design

"Ingestion should scale by metadata growth, not by pipeline code growth."

MIND Pattern · Solution

1 Unified Metadata Table governs N sources — zero code duplication

Problem in brief

  • ▸ One hardcoded pipeline per source
  • ▸ High maintenance after schema changes
  • ▸ Low reuse across environments

MIND Orchestration Process

1 UMT governs N sources
One SQL INSERT to add a new source. The generic engine reads metadata and executes — zero code change.
4 ingestion strategies, runtime-selected
Full · Date-based incremental · Hash-based incremental · Hybrid — chosen per source row in UMT config.
Cloud-agnostic by design
Validated on Azure (ADF + Databricks + ADLS Gen2) and GCP (CockroachDB + Colab) with the same pattern.

RQ2
Pattern Introduction

ETLT / ELTL

Enhanced Hybrid Transformation Patterns

"The core decision is not ETL versus ELT, but transformation timing under explicit forces."

ETLT++/ELTL++ · Solution

Two complementary patterns + an explicit decision matrix

Problem in brief

  • ▸ Teams rely on undocumented hybrid flows
  • ▸ Timing criteria are rarely explicit
  • ▸ Trade-offs remain hard to audit

Decision Matrix — when to choose each pattern

Decision Force ETLT++ ELTL++
Strict quality gate before storagePrimary fitSecondary
Raw data replay and forensicsLimitedPrimary fit
Regulatory compliance + auditabilityHighHigh
Storage efficiency priorityBetterHigher cost
Multi-team raw dataset reuseModerateStrong
ETLT++ — Quality before Load
E → T₁(Quality) → L → T₂(Business) · Contracts at T₁ · Quarantine for rejects · Best for: strict SLAs, regulated sources, strict latency
ELTL++ — Raw fidelity first
E → L(Raw) → T → L(Curated) · Immutable raw for audit & time travel · Multi-team raw reuse · Best for: IoT, finance, forensics

★ Key insight: Both patterns coexist in the same lakehouse — ETLT++ for latency-sensitive streams, ELTL++ for compliance workloads.

ETLT++/ELTL++ · Validation Results

Pattern profiles: architecture, strengths, and adoption contexts

ETLT++ profile — quality-first loading
  • ▸ Stage order: Extract → Transform(quality) → Load → Transform(business)
  • ▸ Contract validation before persistence
  • ▸ Best for regulated, latency-sensitive pipelines
ELTL++ profile — raw-preserving loading
  • ▸ Stage order: Extract → Load(raw) → Transform → Load(curated)
  • ▸ Immutable raw for replay and forensic analysis
  • ▸ Best for audit-heavy and multi-team lakehouse contexts

Pattern usage examples

  • ▸ Databricks DLT: ELTL++ over Bronze/Silver/Gold zones
  • ▸ Azure DEDP: ETLT++ for regulated healthcare pipelines
  • ▸ Finance workloads: ELTL++ for replayability and compliance audits

Pattern architecture reference

★ Interpretation: choose ETLT++ when pre-load quality control dominates; choose ELTL++ when raw fidelity and replay dominate.

RQ3
Pattern Introduction

EADF

Environment-Aware Deployment Framework

"Deployment reliability requires late binding, explicit gates, and immutable provenance."

EADF Pattern · Solution

Late Binding YAML Descriptors + 7-Stage Deterministic Pipeline

Problem in brief

  • ▸ Hardcoded values lead to environment drift
  • ▸ Promotion lacks deterministic audit trails
  • ▸ Data-quality gates are often inconsistent

EADF Workflow

4 YAML Artifacts — the pattern structure

pipeline.yml — zero hardcoded env values, authored once
env-catalog.yml — endpoints/paths per env, bound at deploy time
promotion-gate.yml — quality thresholds (freshness, error rate, schema)
provenance.yml — immutable audit: who, what, from → to, when

Key principle: Same pipeline.yml promotes dev → staging → prod unchanged. Only env-catalog bindings differ. Config drift eliminated by construction.
