01 — Research Motivation

Four Persistent Challenges
in Modern Data Engineering

🌊

Source Heterogeneity

Enterprises ingest from 100+ heterogeneous sources: REST APIs, RDBMS, streams, flat files. No reusable ingestion abstraction exists. Configuration costs consume 10–40% of development time.

⏱️

Transformation Timing

78% of practitioners use ad-hoc hybrid ETL/ELT combinations without formal guidance. Trade-off between early cleaning and late schema flexibility remains unformalized.

☁️

Configuration Crisis

No prior work defines a reusable pattern for multi-environment data pipeline CI/CD. Environment-specific hardcoding causes configuration drift and brittle deployments across dev/staging/prod.

🤖

LLM Reliability Unknown

LLMs promise democratization of data engineering but lack systematic empirical evaluation for production pipeline generation. Capability boundaries and failure modes are uncharted.

04 — Pattern 1

MIND Pattern

Problem: Ingesting 100+ sources requires bespoke connectors. When source schemas evolve, code breaks. Configuration duplication is rampant. No reusable abstraction exists.
Solution — UMT: A Unified Metadata Table centralizes all source definitions (endpoint, schema, schedule, credentials ref). The ingestion engine reads the UMT at runtime — adding a source requires only a new row, no code changes.
  • Forces resolved: reconfigurability vs. security, runtime flexibility vs. governanceLate-binding configuration avoids hardcoded secrets
  • Known uses: UNICC data lakehouse (Azure), multi-cloud GCP pipeline65 → 100+ sources managed without code change

05 — Patterns 2 & 3

ETLT++ / ELTL++ Patterns

Problem: 78% of teams use hybrid ETL+ELT without formal guidance. The correctness-efficiency trade-off (clean early vs. transform late) has no systematic decision framework — each team reinvents the wheel.
ETLT++: Pre-load normalization + post-load enrichment. Use when data quality is critical at landing time, sources are diverse in format, and downstream queries are latency-sensitive.
ELTL++: Load raw → transform on query (late binding). Use when schema evolves rapidly, storage is cheap, and analytical flexibility is the priority.

06 — Pattern 4

EADF Pattern

Problem: CI/CD tools (Jenkins, GitHub Actions) lack data-specific abstractions. MLOps platforms ignore pipeline workflows. IaC handles resources, not artifact promotion. No prior pattern unifies these concerns.
Solution — Late Binding YAML: 4-artifact pattern: pipeline.yml + env-catalog.yml + promotion-gate.yml + provenance.yml. Runtime binding resolves environment values — same artifact promotes through dev → staging → prod.
  • 7-stage promotion: Binding → Build → Validation → Deployment → Smoke Test → Gate → Provenance
  • Complements, not replaces: integrates into existing CI systems
