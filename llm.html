Benchmarking Large Language Models for Data Pipeline Generation

A systematic empirical comparison of 7 state-of-the-art LLMs (GPT-5, Claude 4, Qwen3) across three increasingly complex data engineering pipeline scenarios — measuring fix counts, execution times, code quality, and architectural soundness on Databricks PySpark.

Why LLMs in Data Engineering?

Traditional data pipeline construction requires substantial domain expertise, manual coding effort, and ongoing maintenance overhead. LLMs offer the potential to democratize data engineering — enabling natural language specification of complex transformations and automated code generation across platforms.

However, the systematic evaluation of LLMs across diverse data engineering contexts remained in early stages. This chapter contributes the first controlled empirical assessment of multiple LLMs on identical, production-representative data pipeline tasks.

Research Question RQ4: Can LLMs generate reliable, scalable, and production-ready data engineering code? What are the boundaries of their capability?

Three Pipeline Scenarios (Increasing Complexity)

Complexity: Basic 1 source, 1 API

Air quality + weather API integration: dual ingestion, inner join on timestamp, quality checks on merged dataset, Bronze+Silver layering.

Evaluation Metrics

Fix Count

Number of manual corrections to reach a fully functional pipeline. Lower = better.

Execution Time (s)

Mean runtime over 5 runs on Databricks serverless. Outliers (>2σ) excluded and re-run.

Pylint / PEP8

Static code quality scores (0–10). Note: 0.00 Pylint may indicate parsing failure, not just style.

C&S Score (1–5)

Composite: execution correctness × architectural quality (idiomatic PySpark patterns). Primary metric.

Action Correctness

Expert panel (5 data engineers) rated each action against task spec: FC / PC / IC.

Key Findings & Implications

What this empirical study tells practitioners about LLM selection for data engineering.

🏆

Claude Opus 4: Most Consistent

0–1 fixes across all three pipelines. Fully correct on P1 and P3, stable 9–11s execution. Best choice for correctness-critical enterprise pipelines.

⚡

GPT-5: Optimization Leader

Best execution times (3–9s) once corrected. GPT models prioritize optimization over first-attempt correctness — ideal for performance-sensitive workloads with iteration budget.

🎯

GPT-5 Mini: Best All-Rounder

Zero fixes on all three prompts, fully correct expert rating on P1 and P2. Strong balance of correctness and efficiency across simple and complex tasks.

⚠️

Complexity Cliff Effect

Claude Haiku 3.5: 0 fixes on P1, 1 fix on P2, 8 fixes on P3. Lightweight models hit clear reliability boundaries at multi-source integration complexity.

🔍

Where Failures Concentrate

Errors concentrate in transformation semantics and type handling, not API routing. Fix-count metrics capture this only indirectly — hence the need for expert action correctness evaluation.

🧠

Human Oversight Still Required

LLMs enhance velocity and reduce boilerplate, but architectural decisions, quality-cost trade-offs, and precise cleaning logic still require human expertise. Not a replacement — an accelerator.

Connection to the Design Patterns

The LLM evaluation is not isolated — it directly complements the three formalized patterns. LLMs can generate pattern-conformant code, but require human oversight to enforce pattern constraints consistently.

MIND Pattern

LLMs can generate metadata-driven ingestion code but often miss incremental logic and schema evolution handling — exactly what MIND formalizes.

ETLT/ELTL Patterns

LLMs generate Bronze/Silver architectures well, but contract enforcement, versioning, and SLO monitoring require explicit pattern guidance.

EADF Pattern

LLMs cannot reliably generate multi-environment CI/CD with late binding — EADF's YAML descriptor approach is beyond current zero-shot capability.
